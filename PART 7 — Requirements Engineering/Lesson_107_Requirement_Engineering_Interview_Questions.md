# Section 1 --- Requirement Engineering Fundamentals

## Q1. What is Requirement Engineering?

**Answer:**

Requirement Engineering is the systematic process of gathering,
analyzing, documenting, validating, and managing software requirements
throughout the software development lifecycle.

It ensures that software is built according to stakeholder needs.

------------------------------------------------------------------------

## Q2. Why is Requirement Engineering important?

Requirement Engineering helps:

-   Understand customer expectations
-   Reduce misunderstandings
-   Prevent project failures
-   Control scope
-   Improve software quality

------------------------------------------------------------------------

## Q3. What are the phases of Requirement Engineering?

The main phases are:

``` text
Requirement Elicitation

        ↓

Requirement Analysis

        ↓

Requirement Specification

        ↓

Requirement Validation

        ↓

Requirement Management
```

------------------------------------------------------------------------

# Section 2 --- Requirement Elicitation Questions

## Q4. What is Requirement Elicitation?

Requirement Elicitation is the process of collecting requirements from
stakeholders.

Sources:

-   Customers
-   Users
-   Business teams
-   Domain experts

------------------------------------------------------------------------

## Q5. What are common requirement elicitation techniques?

Techniques include:

-   Interviews
-   Surveys
-   Workshops
-   Observation
-   Brainstorming
-   Prototyping

------------------------------------------------------------------------

## Q6. Why is requirement elicitation difficult?

Because:

-   Users may not clearly express needs
-   Requirements change frequently
-   Different stakeholders have conflicts
-   Hidden requirements exist

------------------------------------------------------------------------

# Section 3 --- Functional and Non Functional Requirements

## Q7. What are Functional Requirements?

Functional Requirements describe what a system should do.

Examples:

-   Login
-   Payment
-   Search
-   Report generation

------------------------------------------------------------------------

## Q8. What are Non Functional Requirements?

Non Functional Requirements describe how a system should perform.

Examples:

-   Performance
-   Security
-   Scalability
-   Reliability

------------------------------------------------------------------------

## Q9. Difference between Functional and Non Functional Requirements?

  Functional         Non Functional
  ------------------ --------------------
  Defines features   Defines quality
  What system does   How system behaves
  Login, payment     Security, speed

------------------------------------------------------------------------

# Section 4 --- SRS Questions

## Q10. What is SRS?

SRS stands for Software Requirement Specification.

It is a document containing detailed software requirements.

------------------------------------------------------------------------

## Q11. What does SRS contain?

SRS contains:

-   Functional requirements
-   Non functional requirements
-   Constraints
-   Assumptions
-   Interfaces

------------------------------------------------------------------------

## Q12. Why is SRS important?

SRS provides:

-   Clear understanding
-   Agreement between stakeholders
-   Development guidance
-   Testing reference

------------------------------------------------------------------------

# Section 5 --- Use Case Questions

## Q13. What is a Use Case?

A Use Case describes interaction between an actor and a system to
achieve a goal.

------------------------------------------------------------------------

## Q14. What are the components of a Use Case?

Components:

-   Actor
-   Goal
-   Preconditions
-   Main Flow
-   Alternate Flow
-   Postconditions

------------------------------------------------------------------------

## Q15. Difference between Use Case and User Story?

  Use Case               User Story
  ---------------------- -------------------
  Detailed interaction   Short requirement
  Scenario based         User value based
  Used in analysis       Used in Agile

------------------------------------------------------------------------

# Section 6 --- User Story Questions

## Q16. What is a User Story?

A User Story describes a requirement from the user's perspective.

Format:

``` text
As a user,
I want feature,
so that benefit.
```

------------------------------------------------------------------------

## Q17. What is INVEST criteria?

A good User Story should be:

-   Independent
-   Negotiable
-   Valuable
-   Estimable
-   Small
-   Testable

------------------------------------------------------------------------

## Q18. What are Acceptance Criteria?

Acceptance Criteria define conditions that must be satisfied before a
User Story is considered complete.

------------------------------------------------------------------------

software systems.

------------------------------------------------------------------------

## Q20. Name common UML diagrams.

-   Use Case Diagram

-   State Diagram

------------------------------------------------------------------------

# Section 8 --- Requirement Traceability Matrix Questions

## Q21. What is RTM?

Requirement Traceability Matrix maps requirements with:

-   Design
-   Code
-   Test Cases

------------------------------------------------------------------------

## Q22. Why is RTM used?

RTM ensures:

-   Complete requirement coverage
-   Better testing
-   Easier change management

------------------------------------------------------------------------

## Q23. What are types of traceability?

Types:

1.  Forward Traceability

Requirement → Test Case

2.  Backward Traceability

Test Case → Requirement

------------------------------------------------------------------------

# Section 9 --- Requirement Validation Questions

## Q24. What is Requirement Validation?

Requirement Validation checks whether requirements are:

-   Correct
-   Complete
-   Consistent
-   Feasible
-   Testable

------------------------------------------------------------------------

## Q25. Difference between Verification and Validation?

  Verification                 Validation
  ---------------------------- ------------------------
  Building product correctly   Building right product
  Checks documents             Checks user needs

------------------------------------------------------------------------

# Section 10 --- Scenario-Based Questions

## Scenario 1

A customer gives unclear requirements. What will you do?

Answer:

-   Ask clarification questions
-   Discuss with stakeholders
-   Create prototypes
-   Document final requirements

------------------------------------------------------------------------

## Scenario 2

Two stakeholders have conflicting requirements.

Answer:

-   Analyze business priority
-   Discuss impact
-   Find a balanced solution

------------------------------------------------------------------------

## Scenario 3

Requirement changes after development starts.

Answer:

-   Perform impact analysis
-   Update documentation
-   Modify backlog and plans

------------------------------------------------------------------------

# Interview Tips

Strong answers should include:

✓ Definition\
✓ Purpose\
✓ Example\
✓ Industry usage

Avoid only memorizing definitions. Interviewers usually test whether you
understand how requirements affect real software development.

# Summary

Requirement Engineering interviews focus on understanding how software
requirements are discovered, documented, validated, and managed. A
strong candidate should explain requirement concepts with practical
examples and show how requirements connect stakeholders, developers,
testers, and final software products.
