# Introduction

Before building software, teams must discover what users and businesses actually need.

This discovery process is called **Requirement Elicitation**.

```text
Stakeholders

      ↓

Requirement Elicitation

      ↓

Software Requirements

      ↓

Development Team
```

---

# What is Requirement Elicitation?

Requirement Elicitation is the process of gathering, discovering, and understanding software requirements from stakeholders.

It answers:

- What problem needs to be solved?
- Who will use the system?
- What features are required?
- What constraints exist?

---

# Why Do We Need Requirement Elicitation?

Without proper elicitation:

```text
Wrong Understanding

        ↓

Wrong Features Built

        ↓

Rework

        ↓

Higher Cost
```

Good elicitation helps:

- Understand user expectations
- Identify hidden requirements
- Reduce misunderstandings
- Define project scope
- Improve software quality

# Requirement Elicitation Techniques

# 1. Interviews

## What is it?

Direct conversation with stakeholders.

Example:

Banking software team interviews customers about mobile banking needs.

Benefits:

- Detailed information
- Direct feedback
- Clarifies confusion

Limitations:

- Time consuming
- Depends on stakeholder availability

---

# 2. Surveys and Questionnaires

Used when many users need to provide feedback.

Example:

Collecting opinions from thousands of customers.

Benefits:

- Large audience
- Fast data collection

---

# 3. Workshops

A group discussion involving:

- Customers
- Developers
- Business analysts
- Experts

Purpose:

Create shared understanding.

```text
Multiple Stakeholders

        ↓

Discussion

        ↓

Common Requirements
```

---

# 4. Observation

Analysts observe users performing tasks.

Useful when users cannot clearly explain workflows.

Example:

Observing hospital staff using existing systems.

---

# 5. Brainstorming

Team generates ideas and possible solutions.

Used during:

- New product creation
- Feature planning

---

# 6. Prototyping

Creating an early model of the system.

Example:

Creating UI screens before development.

Benefits:

- Users visualize requirements
- Early feedback
- Reduces misunderstanding

# Functional vs Non Functional Discovery

During elicitation teams identify:

## Functional Requirements

"What should the system do?"

Examples:

- Login
- Search
- Payment

---

## Non Functional Requirements

"How should the system behave?"

Examples:

- Response time
- Security
- Availability

---

# Challenges in Requirement Elicitation

## 1. Communication Gap

Problem:

Users describe problems differently from developers.

Solution:

Use clear discussions and examples.

---

## 2. Changing Requirements

Business needs change frequently.

Solution:

Maintain flexible requirement management.

---

## 3. Hidden Requirements

Users may forget important needs.

Solution:

Use observation and prototyping.

---

## 4. Conflicting Requirements

Different stakeholders may want different things.

Solution:

Analyze priorities and business value.

- Ask open-ended questions
- Document discussions
- Create prototypes
- Validate requirements
- Prioritize business value
