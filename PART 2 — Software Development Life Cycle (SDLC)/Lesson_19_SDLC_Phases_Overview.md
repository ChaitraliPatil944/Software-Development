# Part 2 — Software Development Life Cycle (SDLC)

> **Objective**
>
> Understand every phase of the Software Development Life Cycle (SDLC), the purpose of each phase, its inputs, outputs, stakeholders, and how all phases work together to build successful software.

---

# Introduction

The SDLC is like a roadmap.

Instead of randomly writing code, software is developed through a sequence of organized phases.

```text
Idea
  |
Requirements
  |
Design
  |
Development
  |
Testing
  |
Deployment
  |
Maintenance
```

Every phase has a specific objective.

# Phase 1 — Requirements

## Purpose

Understand **what** should be built.

### Activities

- Meet stakeholders
- Gather business requirements
- Identify users
- Define project scope

### Inputs

- Business idea
- Customer needs

### Outputs

- BRD (Business Requirements)
- User Stories
- Initial Scope

### Stakeholders

- Customer
- Product Owner
- Business Analyst

---

# Phase 2 — Design

## Purpose

Decide **how** the software will be built.

### Activities

- High-Level Design (HLD)
- Low-Level Design (LLD)
- Database Design
- UI/UX Design
- API Design

### Outputs

- Architecture
- UML Diagrams
- Wireframes

### Stakeholders

- Architect
- UI/UX Designer
- Senior Developers

---

# Phase 3 — Development

## Purpose

Convert the design into working software.

### Activities

- Coding
- Unit Testing
- Code Reviews
- Version Control

### Outputs

- Source Code
- Build Artifacts

### Stakeholders

- Frontend Developers
- Backend Developers
- Database Engineers

---

# Phase 4 — Testing

## Purpose

Verify that the software works correctly.

### Activities

- Functional Testing
- Integration Testing
- Regression Testing
- Performance Testing
- Security Testing

### Outputs

- Bug Reports
- Test Reports
- Release Candidate

### Stakeholders

- QA Engineers
- Developers

---

# Phase 5 — Deployment

## Purpose

Release software to users.

### Activities

- Production deployment
- Configuration
- Smoke testing
- Monitoring

### Outputs

- Live Application

### Stakeholders

- DevOps Engineers
- Operations Team

---

# Phase 6 — Maintenance

## Purpose

Keep software useful after release.

### Activities

- Bug fixes
- Feature enhancements
- Performance tuning
- Security patches

### Outputs

- Updated releases
- Improved software

### Stakeholders

- Developers
- QA
- DevOps
- Support Team

---

# Inputs & Outputs

| Phase | Input | Output |
|------|-------|--------|
| Requirements | Business Idea | Requirements |
| Design | Requirements | Design Documents |
| Development | Design | Source Code |
| Testing | Source Code | Tested Software |
| Deployment | Tested Software | Live Product |
| Maintenance | Live Product | Updated Product |

# Phase Responsibilities

| Phase | Main Goal | Primary Role |
|------|------------|--------------|
| Requirements | Understand problem | Business Analyst |
| Design | Create blueprint | Architect |
| Development | Build software | Developers |
| Testing | Verify quality | QA Engineers |
| Deployment | Release software | DevOps |
| Maintenance | Improve software | Engineering Team |

# Why Every Phase Matters

Skipping requirements:

❌ Wrong software

Skipping design:

❌ Poor architecture

Skipping testing:

❌ Production bugs

Skipping maintenance:

❌ Software becomes obsolete

Every phase supports the next one.

---

# Common Mistakes

- Starting coding before understanding requirements
- Ignoring design documents
- Treating testing as optional
- Deploying without validation
- Ignoring user feedback after release

# Summary

The SDLC divides software development into six interconnected phases. Each phase has a clear purpose, defined activities, specific deliverables, and responsible stakeholders. Following every phase systematically helps teams reduce risk, improve quality, control costs, and deliver software that meets business and user expectations.
