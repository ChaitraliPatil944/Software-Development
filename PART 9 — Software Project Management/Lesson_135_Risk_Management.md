# Introduction

Every software project contains uncertainty.

A project may face:

-   Technical failures
-   Requirement changes
-   Resource problems
-   Security issues
-   Schedule delays

Risk Management helps teams prepare before problems become disasters.

Because apparently software projects enjoy hiding surprises until the
worst possible moment. A little planning is the human attempt to
convince chaos to behave.

``` text
Potential Problem

        ↓

Identify Risk

        ↓

Plan Response

        ↓

Reduce Impact
```

------------------------------------------------------------------------

# What is Project Risk Management?

**Project Risk Management** is the process of identifying, analyzing,
responding to, and monitoring risks that may affect project objectives.

It manages:

-   Scope risks
-   Time risks
-   Cost risks
-   Technical risks
-   Resource risks

------------------------------------------------------------------------

# Why Do We Need Risk Management?

Without risk management:

``` text
Unknown Problem

        ↓

Unexpected Impact

        ↓

Project Delay

        ↓

Failure
```

Risk management helps:

-   Reduce uncertainty
-   Improve decision making
-   Protect project goals
-   Prepare backup plans
-   Improve confidence

------------------------------------------------------------------------

# What is a Project Risk?

A **Risk** is an uncertain event that may negatively or positively
affect project objectives.

Example:

Risk:

    Cloud service failure

Impact:

    Application downtime

# 2. Risk Analysis

Risk analysis evaluates:

-   Probability
-   Impact
-   Severity

------------------------------------------------------------------------

# Risk Probability

Probability means the chance that a risk will happen.

Example:

    Low

    Medium

    High

------------------------------------------------------------------------

# Risk Impact

Impact represents damage caused by the risk.

Example:

    Low Impact

    Medium Impact

    High Impact

------------------------------------------------------------------------

# Risk Priority

A common approach:

    Risk Priority

    =

    Probability × Impact

Example:

    High Probability

    ×

    High Impact

    =

    Critical Risk

# 3. Risk Response Strategies

After analysis, teams decide how to handle risks.

Common strategies:

``` text
Avoid

Mitigate

Transfer

Accept
```

------------------------------------------------------------------------

# 1. Risk Avoidance

Remove the cause of risk.

Example:

Avoid using unstable technology.

------------------------------------------------------------------------

# 2. Risk Mitigation

Reduce probability or impact.

Example:

Perform regular backups.

------------------------------------------------------------------------

# 3. Risk Transfer

Move responsibility to another party.

Example:

Using cloud provider security services.

------------------------------------------------------------------------

# 4. Risk Acceptance

Accept the risk when impact is low.

Example:

Minor UI improvement delays.

------------------------------------------------------------------------

# Risk Register

A **Risk Register** is a document containing identified risks.

Example:

  Risk                Probability   Impact   Response
  ------------------- ------------- -------- -------------------
  Server Failure      High          High     Backup Server
  Developer Leaving   Medium        High     Knowledge Sharing

------------------------------------------------------------------------

# Risk Management in Agile

# Best Practices

-   Maintain risk register
-   Review risks regularly
-   Involve stakeholders
-   Prioritize critical risks

-   Monitor continuously

# Risk Management vs Issue Management

# Summary

Risk Management helps software teams identify and handle uncertainty
before it impacts project success. By analyzing risks, preparing
responses, and continuously monitoring them, organizations reduce
failures and improve project reliability.
