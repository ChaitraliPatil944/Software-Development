# What is the Software Crisis?

The **Software Crisis** refers to the period during the **1960s and 1970s** when software projects became so large and complex that organizations could no longer complete them successfully using informal programming practices.

Projects were:

- Delivered late
- Over budget
- Full of bugs
- Difficult to maintain
- Sometimes abandoned completely

This wasn't a hardware problem.

It was a **software development problem**.

---

# Why Did the Crisis Happen?

Early software was simple.

```
1950

Program
 |
One Programmer
 |
Done
```

Then computers became powerful.

```
1970

Large Company
      |
Thousands of Users
      |
Huge Software
      |
Many Teams
      |
Millions of Lines of Code
```

The methods used for small programs no longer worked.

---

# Root Causes

## 1. Increasing Complexity

Applications grew from hundreds of lines to millions.

Examples:

- Banking
- Airline reservation
- Military systems
- Space programs

---

## 2. No Standard Process

Different developers followed different approaches.

```
Developer A --> Codes directly

Developer B --> Designs first

Developer C --> No documentation

Result:
Confusion
```

---

## 3. Poor Requirement Gathering

Customers often said:

> "Build me a system."

Developers interpreted requirements differently.

Result:

```
Customer Wanted
        |
Developer Understood
        |
Software Built
        |
Customer Unhappy
```

---

## 4. Poor Communication

Large teams struggled to coordinate.

```
Management

  |

Developers

  |

Testers

  |

Customer

(No clear communication)
```

---

## 5. No Testing Strategy

Testing usually happened only after coding finished.

Consequences:

- Thousands of bugs
- Expensive fixes
- Delayed release

---

## 6. Lack of Documentation

When developers left the company, no one understood the code.

```
Developer Leaves

      |

Knowledge Lost

      |

Maintenance Becomes Difficult
```

---

# Symptoms of the Software Crisis

- Missed deadlines
- Budget overruns
- Poor software quality
- Frequent failures
- Difficult maintenance
- Customer dissatisfaction
- High development cost

---

# Real-Life Analogy

Imagine constructing a 100-floor building.

Without:

- Blueprint
- Engineer
- Architect
- Material planning
- Safety inspection

Construction would become chaotic.

Large software projects faced the same problem.

---

# Famous Examples

## Denver Airport Baggage System

Problems:

- Massive delays
- Software failures
- Cost overruns

The project became a classic example of poor planning and complexity.

---

## Ariane 5 Flight 501

A software error caused the rocket to self-destruct shortly after launch.

Lessons:

- Software quality matters.
- Reusing code without proper validation can be dangerous.

---

# How the Industry Responded

The crisis led to the creation of structured practices.

```
Software Crisis
       |
       +--> SDLC
       |
       +--> Waterfall
       |
       +--> Software Design
       |
       +--> Testing
       |
       +--> Documentation
       |
       +--> Project Management
       |
       +--> Version Control
       |
       +--> Agile
       |
       +--> DevOps
```

Every major methodology aims to reduce failure.

# Before vs After

| Before | After |
|---------|--------|
| Just coding | Planned development |
| Little documentation | Complete documentation |
| Individual work | Team collaboration |
| Manual tracking | Project management |
| Testing at end | Continuous testing |
| Difficult maintenance | Maintainable software |

---

# Why It Still Matters

Modern projects are even more complex.

Examples:

- Cloud platforms
- AI systems
- Banking software
- Healthcare systems
- E-commerce

Without engineering discipline, the same problems would return.

---

# Lessons Learned

- Plan before coding.
- Gather requirements carefully.
- Design the system.
- Test continuously.
- Document everything important.
- Manage changes.
- Collaborate effectively.

# Summary

The Software Crisis exposed the limitations of writing software without structured processes. As software systems became larger and more critical, organizations needed engineering practices such as planning, documentation, testing, project management, and lifecycle models. Modern Software Engineering exists largely as a response to the lessons learned during this period.
