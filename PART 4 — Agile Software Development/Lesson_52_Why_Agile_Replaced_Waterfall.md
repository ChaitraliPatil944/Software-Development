# Lesson 52 — Why Agile Replaced Waterfall

# Part 4 — Agile Software Development

> **Objective**
>
> Understand why many organizations moved from the Waterfall Model to Agile, where each approach excels, and how to choose between them.

---

# Introduction

For many years, the Waterfall Model dominated software development.

It worked well when:

- Requirements were stable
- Technology changed slowly
- Customers accepted long delivery cycles

As software became central to business, these assumptions stopped being true.

---

# The Problem with Waterfall

Typical Waterfall timeline:

```text
Requirements
     ↓
Design
     ↓
Development
     ↓
Testing
     ↓
Deployment
```

The customer often sees working software only at the end.

If requirements change midway:

```text
Requirement Changes
        ↓
Design Changes
        ↓
Code Changes
        ↓
Testing Changes
        ↓
Higher Cost
```

---

# Why Businesses Needed Something Different

Modern businesses require:

- Faster releases
- Continuous customer feedback
- Frequent feature updates
- Rapid response to competitors
- Lower risk

These needs encouraged the adoption of Agile.

---

# How Agile Solves These Problems

```text
Sprint 1
   ↓
Working Software
   ↓
Feedback
   ↓
Sprint 2
   ↓
Working Software
   ↓
Feedback
```

Small releases reduce uncertainty and allow teams to adapt.

---

# Waterfall vs Agile

| Waterfall | Agile |
|-----------|--------|
| Sequential | Iterative |
| Fixed requirements | Changing requirements |
| Large release | Frequent releases |
| Testing after development | Continuous testing |
| Heavy upfront planning | Continuous planning |
| Limited customer interaction | Continuous collaboration |
| High cost of change | Lower cost of change |

---

# Example

## Food Delivery App

### Waterfall

- Plan everything
- Develop for 10 months
- Release once

Problem:

Customers now want live tracking and digital wallets.

Changes become expensive.

### Agile

Sprint 1

- Login

Sprint 2

- Restaurant Search

Sprint 3

- Cart

Sprint 4

- Payments

Live tracking can be added in a future sprint.


```text

   ↕
```

---

# Risk Comparison

Waterfall:

- Late discovery of defects
- Late validation
- Requirement risk

Agile:

- Frequent validation
- Smaller releases
- Faster issue detection

---

# Does Agile Completely Replace Waterfall?

No.

Waterfall is still valuable for:

- Government projects
- Defense systems
- Medical compliance
- Fixed contractual requirements

Agile is ideal for:

- SaaS
- Web applications
- Mobile apps
- Cloud platforms
- AI products

# Advantages of Agile Over Waterfall

- Faster value delivery
- Better customer satisfaction
- Easier requirement changes
- Continuous testing
- Continuous improvement
- Better visibility

---

# Situations Where Waterfall Is Better

- Fixed scope
- Strict compliance
- Predictable timelines
- Extensive documentation
- Minimal requirement changes

---

# Common Misconceptions

❌ Agile is always better.

✔ The right model depends on the project.

❌ Waterfall is obsolete.

✔ It remains appropriate for many regulated industries.

# Summary

Agile became popular because it addresses the limitations of traditional sequential development in fast-changing business environments. By delivering software in small increments, encouraging customer collaboration, and embracing change, Agile enables teams to build products that evolve with user needs. Waterfall, however, continues to be the preferred choice for projects with stable requirements, strict compliance, and predictable execution.
