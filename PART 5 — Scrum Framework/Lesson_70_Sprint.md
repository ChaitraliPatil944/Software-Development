# Introduction

Scrum does not build the entire product at once.

Instead, work is divided into short development cycles called **Sprints**.

```text
Large Product Goal

        ↓

Sprint 1 → Feature Set 1
Sprint 2 → Feature Set 2
Sprint 3 → Feature Set 3

        ↓

Complete Product
```

---

# What is a Sprint?

A **Sprint** is a fixed time period during which a Scrum Team creates a usable and potentially releasable product increment.

A Sprint contains:

- Planning
- Development
- Testing
- Review
- Improvement

---

# Sprint Duration

Typical Sprint duration:

```text
1 Week
2 Weeks
3 Weeks
4 Weeks
```

Most organizations use:

```text
2 Week Sprint
```

because it balances:

- Fast feedback
- Enough development time

---

# Why Do We Need Sprints?

Before Scrum:

```text
Requirement
      ↓
Months of Development
      ↓
Final Release
      ↓
Feedback
```

Problem:

Changes become expensive.

---

With Sprints:

```text
Sprint
 ↓
Working Feature
 ↓
Feedback
 ↓
Improvement
```

Benefits:

- Early feedback
- Reduced risk
- Faster delivery
- Better adaptability

---

# Sprint Lifecycle

```text
Sprint Start

      ↓

Sprint Planning

      ↓

Development + Testing

      ↓

Daily Scrum

      ↓

Sprint Review

      ↓

Sprint Retrospective

      ↓

Next Sprint
```

---

# Sprint Goal

Every Sprint has a clear objective.

Example:

Product:

Food Delivery Application

Sprint Goal:

"Enable customers to search and select restaurants."

Sprint Work:

- Restaurant API
- Search UI
- Filter functionality
- Testing

---

# Sprint Components

## 1. Sprint Planning

The team decides:

- Sprint Goal
- Selected backlog items
- Implementation approach

---

## 2. Sprint Backlog

Contains the work selected for the Sprint.

Example:

```text
Sprint Goal:
Build Login System

Tasks:

✓ Create Login API
✓ Design Login Page
✓ Add Validation
✓ Test Authentication
```

---

## 3. Development

Team builds the selected features.

Activities:

- Coding
- Testing
- Integration
- Review

---

## 4. Daily Scrum

Short daily meeting.

Purpose:

- Synchronize progress
- Identify blockers

---

## 5. Sprint Review

The team demonstrates completed work.

Output:

- Stakeholder feedback
- Backlog updates

---

## 6. Sprint Retrospective

The team discusses:

- What went well?
- What needs improvement?

---

# Sprint Rules

## Rule 1 — Fixed Duration

Sprint duration does not change frequently.

---

## Rule 2 — Sprint Goal Protection

The team focuses on achieving the Sprint Goal.

---

## Rule 3 — Quality Cannot Be Reduced

The team should not sacrifice quality to complete more work.

---

## Rule 4 — Scope Can Be Negotiated

If necessary, work can be adjusted while protecting the Sprint Goal.

# Sprint vs Traditional Development

| Sprint | Traditional Development |
|---|---|
| Short cycle | Long cycle |
| Frequent delivery | Late delivery |
| Continuous feedback | Feedback at end |
| Adaptable | Less flexible |

---

# Sprint vs Iteration

| Sprint | Iteration |
|---|---|
| Scrum term | General Agile term |
| Has defined events | May vary |
| Has Sprint Goal | May not have goal |

---

# Sprint Success Factors

Successful Sprints require:

- Clear Sprint Goal
- Prioritized backlog
- Team collaboration
- Realistic estimation
- Continuous testing
- Stakeholder feedback

---

# Common Mistakes

❌ Adding unlimited work during Sprint.

✔ Protect Sprint focus.

---

❌ Skipping Sprint Review.

✔ Collect feedback regularly.

---

❌ Treating Sprint as a deadline only.

✔ Sprint is a learning and delivery cycle.

# Summary

A Sprint is the foundation of Scrum execution. It provides a fixed time-box where teams plan, build, test, review, and improve continuously. By delivering small increments instead of waiting for a final release, Sprints help organizations reduce risk, gather feedback quickly, and adapt to changing customer needs.
