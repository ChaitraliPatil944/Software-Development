# Lesson 75 — Sprint Backlog

# Part 5 — Scrum Framework

> **Objective**
>
> Understand the Sprint Backlog, its purpose, ownership, structure, relationship with Sprint Goals, and how Scrum teams use it to manage work during a Sprint.

---

# Introduction

A Scrum Team needs a clear plan for the current Sprint.

The **Sprint Backlog** provides that plan.

```text
Product Backlog
        ↓
Sprint Planning
        ↓
Sprint Backlog
        ↓
Development Work
        ↓
Product Increment
```

---

# What is Sprint Backlog?

The **Sprint Backlog** is a collection of:

- Product Backlog Items selected for the Sprint
- The plan for delivering them
- Technical tasks required to complete the work

It represents the team's commitment for the Sprint.

---

# Why Do We Need Sprint Backlog?

Without Sprint Backlog:

```text
Many Requirements
        ↓
No Clear Focus
        ↓
Poor Coordination
        ↓
Sprint Failure
```

With Sprint Backlog:

```text
Sprint Goal
      ↓
Selected Work
      ↓
Clear Execution Plan
      ↓
Successful Delivery
```

---

# Sprint Backlog Ownership

The Sprint Backlog is owned by:

```text
Developers
```

Developers decide:

- How work is completed
- Task breakdown
- Technical approach
- Daily adjustments

---

# Components of Sprint Backlog

The Sprint Backlog contains:

## 1. Sprint Goal

The objective of the Sprint.

Example:

"Enable customers to complete online payments."

---

## 2. Selected Product Backlog Items

User stories chosen for the Sprint.

Example:

```text
✓ Payment Feature
✓ Checkout Improvement
✓ Transaction History
```

---

## 3. Tasks

Technical work required.

Example:

Payment Feature:

```text
Tasks:

✓ Create Payment API
✓ Integrate Gateway
✓ Build Payment UI
✓ Write Tests
```

---

# Sprint Backlog Structure

Example:

```text
Sprint Goal:

Implement User Authentication


User Story:

User Login


Tasks:

1. Database Setup
2. API Development
3. UI Creation
4. Validation
5. Testing
```

# Sprint Backlog Workflow

```text
Product Backlog

      ↓

Sprint Planning

      ↓

Selected Stories

      ↓

Task Breakdown

      ↓

Sprint Backlog

      ↓

Daily Updates

      ↓

```

---

# Updating Sprint Backlog

The Sprint Backlog is not a fixed document.

Developers update it as they learn more.

Changes may include:

- Adding tasks
- Removing unnecessary tasks
- Re-estimating remaining work

---

Sprint Goal:

"Allow users to place orders."

Sprint Backlog:

User Stories:

---

# Sprint Backlog vs Product Backlog

| Sprint Backlog | Product Backlog |
|---|---|
| Current Sprint work | Entire product work |
| Owned by Developers | Owned by Product Owner |
| Short-term plan | Long-term roadmap |
| Changes during Sprint | Continuously refined |

---

# Sprint Backlog vs Task List

A simple task list:

```text
✓ Code
✓ Test
✓ Deploy
```

Sprint Backlog:

```text
Sprint Goal
      +
User Stories
      +
Tasks
      +
Execution Plan
```

Sprint Backlog provides context.

---

# Sprint Backlog and Daily Scrum

During Daily Scrum:

Developers inspect:

- Sprint progress
- Remaining work
- Possible adjustments

Example:

```text

Today:
Complete UI

Blocker:
Waiting for database access
```

# Common Mistakes

## Mistake 1

Product Owner controlling Sprint Backlog.

Solution:

Developers own execution decisions.

---

## Mistake 2

Adding unrelated work.

Solution:

Protect Sprint Goal.

---

## Mistake 3

Not updating Sprint Backlog.

Solution:

Keep it current throughout Sprint.

# Summary

The Sprint Backlog is the execution plan created during Sprint Planning. It contains the Sprint Goal, selected Product Backlog Items, and tasks required to deliver the product increment. Owned by Developers, it provides transparency, focus, and flexibility throughout the Sprint while allowing teams to adapt as they learn.
